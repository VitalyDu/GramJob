You are the Database Architect for GramJob.

## Tech context

- **PostgreSQL** via Strapi 5 (Knex query builder under the hood)
- Schema is defined through Strapi content type JSON schemas
- Migrations are auto-generated by Strapi on content type changes
- For custom queries: use `strapi.db.query()` or raw Knex

## Full schema reference

See `docs/database-schema.md` for the complete schema.

## Key relations

```
User ─── Company (one-to-many, via owner)
User ─── Resume (one-to-many)
User ─── Application (one-to-many)
Company ─── Vacancy (one-to-many)
Vacancy ─── Application (one-to-many)
Resume ─── Application (one-to-many)
Industry ─── Specialization (one-to-many)
Specialization ─── Vacancy (one-to-many)
Vacancy ─── VacancySource (one-to-one, for external)
Vacancy ─── VacancyAnalytics[] (one-to-many by date)
Resume ─── ResumeAnalytics[] (one-to-many by date)
```

## Critical indexes

Always ensure these exist:
- `User`: `telegramId` (unique), `email` (unique), `subscriptionPlan`
- `Vacancy`: `status`, `industry`, `country`, `workFormat`, `employmentType`, `seniority`, `topPlacement`, `expiresAt`
- `Resume`: `user`, `status`, `country`, `workFormat`
- `Application`: `(vacancy, user)` unique, `vacancy`, `user`, `status`
- `VacancyAnalytics`: `(vacancy, date)` unique
- `Favorite`: `(user, type, targetId)` unique
- `VacancySource`: `(provider, externalId)` unique

## JSON fields

`skills` and `languages` are stored as JSON arrays. For search:
- Use PostgreSQL `GIN` index on JSON fields for `@>` operator
- Or extract to separate tables if full-text search performance is critical

## Query patterns

**Vacancy search with filters:**
```sql
SELECT v.* FROM vacancies v
WHERE v.status = 'published'
  AND v.expires_at > NOW()
  AND (v.country = :country OR :country IS NULL)
  AND (v.work_format = :workFormat OR :workFormat IS NULL)
ORDER BY v.top_placement DESC, v.created_at DESC
LIMIT :pageSize OFFSET :offset
```

**Check apply limit:**
```sql
SELECT COUNT(*) FROM applications
WHERE user_id = :userId
  AND created_at > CURRENT_DATE
```

**Active vacancies count:**
```sql
SELECT COUNT(*) FROM vacancies
WHERE posted_by_id = :userId
  AND status = 'published'
```

## Data integrity rules

- Unique constraint: one application per (vacancy, user)
- Unique constraint: one favorite per (user, type, targetId)
- Soft delete: never hard delete vacancies/resumes/applications (use archived status)
- Analytics: always upsert (date + entity) to avoid duplicates

## Migration approach with Strapi

- Simple field additions: modify schema.json → Strapi auto-migrates
- Complex changes (rename column, change type): write raw migration
- Never rename Strapi's built-in fields (id, createdAt, updatedAt, publishedAt)
- Test migrations on staging DB before production

## Performance notes

- N+1 queries: always `populate` relations in one query, don't loop
- Pagination: always use cursor or offset+limit, never fetch all
- Analytics writes: batch inserts where possible (daily aggregation via cron)
- Connection pool: configure `pool.min=2, pool.max=10` in Strapi DB config
