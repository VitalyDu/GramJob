## 🔴 Критические проблемы

### C1. Поиск пользователя через Documents API по числовому id — сломаны публикация, отклики и доступ к базе резюме

В Strapi 5 `documentId` пользователя — случайная строка, **не** числовой `id`. Проверено на живой БД:

```
 id |       document_id
----+--------------------------
  2 | j4av1csydi05308linm6yxi0
  1 | gqmwdyvuytn2fyhrjfdvb272
```

Паттерн `strapi.documents('plugin::users-permissions.user').findOne({ documentId: String(userId) })` всегда возвращает `null`. Затронуты:

| Место                                                                                            | Следствие                                                                                         |
| ------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------- |
| `backend/src/api/vacancy/services/credit-service.ts:51` (`checkAndConsumeVacancyCredit`)         | `POST /vacancies/:id/publish` → 500 «User not found» — **публикация вакансий не работает**        |
| `backend/src/api/vacancy/services/credit-service.ts:95` (`checkAndConsumeBoost`)                 | `POST /vacancies/:id/boost` → 500                                                                 |
| `backend/src/api/application/services/apply-credit-service.ts:50` (`checkAndConsumeApplyCredit`) | `POST /applications` → 500 — **отклики не работают**                                              |
| `backend/src/api/resume/policies/requires-max-plan.ts:17`                                        | `fullUser = null` → policy возвращает `false` → `GET /resumes` даёт **403 всем, включая Max/VIP** |

Юнит-тесты это не ловят, потому что Strapi замокан. **Фикс:** заменить на `strapi.db.query('plugin::users-permissions.user').findOne({ where: { id: userId }, select: [...] })`.

### C2. Платёжный webhook fail-open: без секрета принимает поддельные платежи

`backend/src/api/payment/controllers/telegram-webhook.ts:37-45` — секрет проверяется **только если** задан `TELEGRAM_WEBHOOK_SECRET`. Если переменная не установлена (или опечатка в имени), любой может отправить `POST /api/telegram/webhook` с фейковым `successful_payment` и бесплатно активировать подписку / начислить кредиты любому userId.

**Фикс:** fail-closed — если секрет не задан, отклонять запрос (или падать на старте, если задан `TELEGRAM_BOT_TOKEN`, но нет секрета).

### C3. Нет идемпотентности платежей и нет таблицы платежей вообще

`handleSuccessfulPayment` (`telegram-webhook.ts:76-120`) не сохраняет и не проверяет `telegram_payment_charge_id`. Telegram **ретраит** webhook при таймауте/5xx → один платёж может быть обработан несколько раз (двойное начисление кредитов / продление). Также в системе нет content type Payment — нет аудита транзакций, невозможно разбирать споры и возвраты.

**Фикс:** таблица `payment` с unique-ключом по `telegram_payment_charge_id`; перед начислением — insert, при конфликте — skip.

### C4. Перепутан порядок SQL-параметров в FTS-поиске при активных блокировках

`backend/src/api/vacancy/services/vacancy.ts:91-101`: плейсхолдеры в SQL идут в порядке `WHERE tsquery → extraFilters → ORDER BY tsquery → LIMIT → OFFSET`, а параметры переданы как `[searchQuery, searchQuery, ...extraParams, limit, offset]`. Когда у пользователя есть блокировки (`blockSql = "AND posted_by_id NOT IN (?)"`), в `NOT IN` подставляется **текст поиска**, а id заблокированного юзера уходит в `ts_rank` → ошибка типов Postgres / неверные результаты. Поиск с блокировками падает.

**Фикс:** `[searchQuery, ...extraParams, searchQuery, limit, offset]` (count-запрос корректен).

### C5. Названия permissions не совпадают с контроллером — покупка пакетов даёт 403

`backend/src/scripts/seed-permissions.ts:47-48` включает `api::payment.payment.buyVacancyPack` / `buyApplyPack`, но handlers в `payment/routes/payment.ts` называются `payment.vacancyPack` / `payment.applyPack`. Реальные actions — `api::payment.payment.vacancyPack`/`applyPack` — **не включены** → `POST /payments/vacancy-pack` и `/payments/apply-pack` всегда 403. Также `api::payment.payment.handleWebhook` в PUBLIC_PERMISSIONS — мёртвая запись (реальный action `api::payment.telegram-webhook.handle`, но route и так `auth: false`).

## 🟠 Высокий приоритет

### H1. План VIP получает лимиты Free

`PLAN_LIMITS` (`credit-service.ts:3-7`) и `APPLY_PLAN_LIMITS` (`apply-credit-service.ts:3-7`) не содержат ключ `vip` → fallback на free: VIP-работодатель (999+499 Stars/мес) получает **3 вакансии/мес, 3 буста/день, 3 отклика/день** вместо лимитов Max. По бизнес-правилам VIP = «как Max».

### H2. Boost не влияет на выдачу — кредит тратится впустую

`vacancy.ts` (controller) `boost`: списывает буст и делает `update({ data: {} })`, «bump updatedAt». Но сортировка выдачи (`sortMap.relevance = 'topPlacement:desc,createdAt:desc'`, FTS — `ts_rank, top_placement, created_at`) **нигде не использует updatedAt**. Пользователь платит бустами за ноль эффекта. По спецификации boost должен поднимать в выдаче + блок рекомендаций.

### H3. Boost-кредиты из Vacancy Pack никогда не начисляются

- Invoice обещает «N вакансий + N буст-кредитов» (`payment.ts:66`), но webhook начисляет только `vacancyCredits` (`telegram-webhook.ts:92-101`).
- У пользователя вообще **нет поля `boostCredits`** (`extensions/users-permissions/content-types/user/schema.json`).
- `checkAndConsumeBoost` учитывает только дневной лимит плана — купленные бусты негде хранить и нельзя потратить.

### H4. Telegram Login Widget можно реплеить бесконечно

`validateWebWidget` (`telegram-validation.ts:48-65`) проверяет HMAC, но **не проверяет `auth_date`** (в отличие от `validateInitData`, где TTL 24 ч). Перехваченный однажды payload виджета даёт вход в аккаунт навсегда. Фикс: проверять свежесть `auth_date` (Telegram рекомендует ~1 день максимум).

### H5. Подписки: длительность и продление считаются неверно

`activateSubscription` (`subscription-service.ts:32-48`):

- жёстко `calculateExpiresAt(30)` — поле `durationDays` из SubscriptionPlan игнорируется;
- продление **перезаписывает** дату от «сейчас», а не от текущего `subscriptionExpiresAt` → продлившись за неделю до конца, пользователь теряет неделю;
- покупка VIP ставит `subscriptionPlan='vip'` с новым сроком 30 дней — сроки Max и VIP-надстройки не разделены (по бизнес-модели VIP — надстройка над активным Max за +499);
- после истечения VIP пользователь падает сразу на `free`, хотя Max мог быть оплачен позже.

### H6. Rejected-компанию невозможно отправить на повторную модерацию

`canSubmit` (`company-utils.ts:10-12`) разрешает submit только из `draft`, а `update` разрешён для `draft|rejected`, но статус при этом не меняется. Итог: компания со статусом `rejected` — тупик (только удалить и создать заново). По схеме статусов резюме/вакансий rejected → повторная подача допускается; для компаний флоу сломан.

### H7. generateMetadata накручивает просмотры вакансий

`frontend/src/app/vacancies/[id]/page.tsx:11-14` — серверный fetch `GET /vacancies/:id` для метаданных. Этот endpoint инкрементирует `views`/`uniqueViews` → каждый рендер метаданных (и каждая ревалидация ISR) добавляет просмотр с IP сервера. Метрики и аналитика (Sprint 7) завышены. Фикс: отдельный лёгкий endpoint без счётчика или query-параметр `?noCount=1` для SSR.

### H8. search_vector новой вакансии — гонка с транзакцией

`vacancy/content-types/vacancy/lifecycles.ts:56-63` — `updateSearchVector` вызывается без await «после коммита», но фактически raw-запрос уходит по **другому соединению** и может выполниться до коммита транзакции Strapi → `UPDATE ... WHERE id = ?` затронет 0 строк, вакансия останется без `search_vector` (не находится поиском). Комментарий в коммите cbcb946 подтверждает, что это workaround дедлока, но замена await на fire-and-forget меняет гонку, а не устраняет её. Надёжный фикс: пересчитывать вектор триггером Postgres (`tsvector_update_trigger` / генерируемая колонка).

## 🟡 Средний приоритет

### M1. Кредиты и счётчики: race conditions

- `addCredits` (`subscription-service.ts:50-73`) — read-then-write без транзакции; параллельные вебхуки теряют начисления. Фикс: `UPDATE ... SET x = x + ?`.
- `checkAndConsumeVacancyCredit` — списание кредита и подсчёт месячного лимита неатомарны (двойной publish при гонке). Известное MVP-ограничение, но фикс планировался в Sprint 6 и не сделан.
- `application.create` — проверка дубликата и вставка без unique-констрейнта `(vacancy, user)` → возможны дубли откликов; кредит списывается **до** создания записи — при падении create кредит сгорает.

### M2. In-memory состояние не переживает рестарт и не масштабируется

- дневные лимиты бустов (`credit-service.ts:20`) и откликов (`apply-credit-service.ts:19`) обнуляются при каждом рестарте/деплое;
- `viewedIPs` в vacancy controller — неограниченный рост (утечка памяти) + сброс уникальных просмотров;
- rate-limit (`middlewares/rate-limit.ts`) — на процесс.
  При запуске >1 инстанса всё это тихо ломается. (Частично задокументировано как MVP-ограничение — но комментарии «replaced in Sprint 6» неверны, Sprint 6 этого не сделал.)

### M3. `proxy: true` не включён в server.ts

За reverse-proxy (любой прод-деплой) `ctx.ip` = IP прокси: **все пользователи делят один rate-limit** (100 req/мин на всех; авторизация — 10/мин на всех) и uniqueViews считается по одному IP. Фикс: `proxy: true` в `config/server.ts` + доверенный заголовок.

### M4. GET /resumes/:id публичный — обходит гейт Max-подписки

`resume/routes/resume.ts:29-33` (`auth: false`) + public permission в seed. Полное резюме (опыт, образование, всё кроме контактов) доступно любому по documentId без Max-плана. Листинг закрыт policy, но карточка — нет. Решить осознанно: либо auth+Max для не-владельцев, либо задокументировать как публичную страницу.

### M5. `toSlug` не поддерживает кириллицу

`company-utils.ts:1-8` вырезает всё, кроме `[a-z0-9]` → «Рога и Копыта» даёт пустой slug. Для RU-аудитории почти все компании получат slug вида `-1`, `-2`. Нужна транслитерация.

### M6. TELEGRAM_WEBHOOK_URL в .env.example без префикса `/api`

Route — `/api/telegram/webhook`, а пример — `https://api.gramjob.com/telegram/webhook` → вебхук из коробки регистрируется на 404-URL.

### M7. fetchMe разлогинивает при любой ошибке

`frontend/src/stores/AuthStore.ts:111-120` — сеть моргнула или бэкенд вернул 500 → `logout()` и потеря сессии. Разлогинивать стоит только на 401/403.

### M8. Module-level `authToken` в API-клиенте

`frontend/src/services/api.ts:14` — глобальная переменная модуля. Сейчас её выставляет только клиент, но любой будущий вызов `setAuthToken` в серверном коде (SSR) утечёт токен между запросами разных пользователей. Как минимум — предохранительный комментарий/проверка `typeof window`.

### M9. SEO-требование SSR/ISR фактически не выполнено

Публичные страницы (`/vacancies`, `/vacancies/[id]`, `/companies`, `/resumes`) рендерят контент целиком на клиенте (`*Client.tsx` + useEffect-fetch); на сервере только `<h1>` и metadata. Контента нет в HTML → цели «SEO на всех публичных страницах, Lighthouse > 90» не достигаются. Нужен серверный fetch данных + гидрация.

### M10. Аналитика: растущая стоимость cron и отсутствие unique-индекса

`config/cron-tasks.ts:309+` — для каждой вакансии ежедневно читается **вся история** её analytics-строк (суммирование для дельты) → O(N×дни), деградация со временем. Нет unique `(vacancy, date)` — дубли возможны при ручном запуске. Также `resume-analytics.uniqueViews = deltaViews` — фиктивные «уникальные».

### M11. Просмотр резюме/вакансии владельцем учитывается в статистике

`resume.findOne` и `vacancy.findOne` инкрементируют views всем, включая владельца и повторные заходы (для резюме unique вообще не считается).

### M12. Уведомления модерации — только approve

Lifecycle-хуки company/resume шлют `moderation_approved`, но `moderation_rejected` не шлётся нигде (нет хука на `status=rejected`), у vacancy publish — TODO. Кандидат/работодатель не узнаёт об отклонении. (Запланировано на Sprint 8 — убедиться, что попадёт в scope.)

## ⚪ Низкий приоритет / замечания

1. **DELETE /vacancies/:id выполняет archive** (`vacancy/routes/vacancy.ts:72-79`) — семантика обманчива, фронтовый `deleteVacancy` на самом деле архивирует. Либо переименовать, либо реализовать удаление для draft.
2. **Favorite/Block не проверяют существование цели** — можно добавить в избранное несуществующий documentId (в списке будет `entity: null`), заблокировать несуществующий userId.
3. **SavedSearch без лимита на пользователя** — можно насоздавать тысячи записей; cron обходит до 5000 за прогон.
4. **Report**: нет ограничения длины `comment` и анти-спама, кроме общего rate-limit.
5. **`filters` SavedSearch не валидируются по структуре** при создании — cron устойчив (собирает фильтры сам), но мусор в БД возможен.
6. **`parseInvoicePayload`** не валидирует `planCode` по списку допустимых (webhook активирует любой `planCode as any`; сейчас источник — свой же сервер, но после C2/C3 стоит ужесточить).
7. **Rate-limit заголовков и `koa-ratelimit` memory-driver** — на Bun/кластере поведение не определено; для прода нужен Redis-driver.
8. **`telegram-auth` create user**: гонка двух первых логинов одного telegramId → второй упадёт на unique-констрейнте (обработать 409/ретраем).
9. **Notification schema** — нет индекса по `(user, isRead)`; списки и unread-count будут медленными на объёме.
10. **Vacancy expire cron** обновляет по одной вакансии в цикле (до 1000 update'ов в час) — можно одним updateMany + пачкой уведомлений.
11. **CSP `img-src`** не включает продовый S3-endpoint (только `localhost:9000` и `*.gramjob.com`) — при выносе бакета на другой домен картинки перестанут грузиться в админке.
12. **Вакансия: `update` published-вакансии сбрасывает в draft с `expiresAt=null`**, и повторная публикация требует ещё один кредит — вероятно by design (модерация правок), но UX-ловушка: пользователь может «потерять» кредит из-за исправления опечатки. Стоит подтвердить у продукта.
13. **`subscription_expiring` за 7 дней** шлётся по окну `(6,7]` дней — если cron не отработал (даунтайм), уведомление пропускается навсегда (нет флага «отправлено»).
