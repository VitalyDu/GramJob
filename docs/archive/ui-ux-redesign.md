# GramJob — Полный редизайн UI/UX и функциональные улучшения

## Цель

Выполнить полный редизайн веб-приложения GramJob, сохранив существующую бизнес-логику.

Основная цель — создать современный, чистый и полностью адаптивный интерфейс на основе **shadcn/ui**, улучшить пользовательский опыт, привести весь интерфейс к единому стилю и реализовать функциональные изменения, описанные ниже.

Редизайн не должен изменять существующую бизнес-логику, кроме явно указанных изменений.

---

# 1. Полный редизайн интерфейса

Необходимо полностью переработать пользовательский интерфейс приложения.

Требуется выполнить редизайн:

- всех страниц;
- всех компонентов;
- всех форм;
- всех модальных окон;
- всех таблиц;
- всех карточек;
- всей навигации;
- поиска;
- фильтров;
- пустых состояний (Empty States);
- состояний загрузки (Loading States);
- состояний ошибок (Error States).

Итоговый интерфейс должен выглядеть как современный SaaS-продукт, а не MVP.

---

# 2. Использование shadcn/ui

Во всём проекте необходимо максимально использовать готовые компоненты **shadcn/ui**.

В частности использовать:

- Button
- Input
- Select
- Combobox
- Dialog
- Drawer / Sheet
- Popover
- Dropdown Menu
- Card
- Badge
- Tabs
- Tooltip
- Table
- Pagination
- Skeleton
- Toast
- Alert

Не создавать собственные реализации компонентов, если аналог уже существует в shadcn/ui.

Для стилизации использовать Tailwind CSS согласно рекомендациям shadcn/ui.

---

# 3. Адаптивность

Приложение должно быть полностью адаптивным.

Поддерживаемые разрешения:

- Mobile
- Tablet
- Desktop
- Large Desktop

На мобильных устройствах не должно появляться горизонтального скролла.

Все страницы должны корректно перестраиваться под размер экрана.

---

# 4. Единый контейнер

В настоящий момент страницы используют разные размеры контейнеров.

Необходимо привести весь проект к единому контейнеру.

Все страницы должны использовать одинаковую максимальную ширину и единые правила внутренних отступов.

---

# 5. Нижняя навигация (Bottom Navigation)

Для мобильной версии веб-приложения необходимо реализовать нижнюю навигацию.

Пункты меню:

- Вакансии
- Отклики
- Избранное
- Кабинет

Требования:

- фиксируется в нижней части экрана;
- отображается только на мобильных устройствах;
- активный раздел должен подсвечиваться;
- использовать иконки;
- обеспечить плавные переходы между разделами.

На десктопной версии существующую навигацию менять не требуется, если только это не улучшает общий UX.

---

# 6. Страница «Кабинет»

Страница «Кабинет» в настоящее время отсутствует.

Необходимо самостоятельно спроектировать и реализовать полноценный личный кабинет пользователя.

Claude должен самостоятельно определить наиболее подходящую структуру страницы, основываясь на существующей бизнес-логике проекта.

Интерфейс должен выглядеть как современная панель управления пользователя и предоставлять быстрый доступ к наиболее важным разделам и действиям.

Личный кабинет должен одинаково хорошо подходить как работодателю, так и кандидату.

---

# 7. Редизайн фильтров вакансий

Полностью переработать интерфейс фильтрации вакансий.

Требования:

- сохранить все существующие фильтры;
- улучшить удобство использования;
- улучшить визуальную иерархию;
- улучшить расположение элементов;
- обеспечить корректную адаптивность.

### Desktop

Фильтры должны оставаться отображёнными на странице.

### Mobile

Фильтры должны открываться в модальном окне (Drawer / Sheet / Dialog).

На самой странице должны отображаться только:

- строка поиска;
- кнопка «Фильтры».

По нажатию кнопки открывается модальное окно со всеми существующими фильтрами.

---

# 8. Изменение процесса публикации вакансии

Изменить процесс создания вакансии.

Текущее поведение:

Создание вакансии → Draft

Новое поведение:

Создание вакансии → Moderation

После создания вакансия должна автоматически отправляться на модерацию без сохранения в статусе Draft.

---

# 9. Изменение процесса редактирования вакансии

Изменить процесс редактирования вакансии.

Текущее поведение:

Редактирование → Draft

Новое поведение:

Редактирование → Moderation

После любого изменения вакансия должна повторно отправляться на модерацию независимо от её предыдущего статуса.

---

# 10. Карточки вакансий

Дополнить карточки вакансий новой информацией.

Необходимо отображать:

- 👁 количество просмотров;
- 📨 количество откликов.

Эти показатели должны отображаться непосредственно на карточке вакансии в списке вакансий.

При этом карточка должна оставаться компактной и легко читаемой.

---

# 11. Главная страница

Полностью переработать главную страницу.

Главная страница должна стать значительно более современной, привлекательной и интерактивной.

Claude самостоятельно принимает решения по улучшению пользовательского опыта.

Допускается добавить:

- современный Hero-блок;
- качественные CTA-блоки;
- статистику сервиса;
- подборки вакансий;
- подборки компаний;
- анимации;
- улучшенную типографику;
- улучшенную структуру страницы;
- современные визуальные эффекты.

Основная цель — повысить качество восприятия продукта.

---

# 12. Общие UX-улучшения

Во время выполнения редизайна необходимо улучшить общий пользовательский опыт.

Допускается:

- уменьшить количество лишних действий;
- улучшить навигацию;
- улучшить читаемость интерфейса;
- улучшить расположение элементов;
- улучшить доступность (Accessibility);
- улучшить Empty States;
- улучшить Loading States;
- повысить единообразие интерфейса.

Claude может самостоятельно принимать разумные UX-решения, если они не изменяют бизнес-логику проекта.

---

# 13. Качество реализации

При реализации придерживаться современных практик разработки React и Next.js.

Требования:

- переиспользуемые компоненты;
- чистая архитектура;
- отсутствие дублирования кода;
- корректная типизация TypeScript;
- адаптивная верстка;
- доступный интерфейс (Accessibility);
- удобная структура компонентов.

---

# 14. Бизнес-логика

Не изменять существующую бизнес-логику проекта.

Исключение составляет только изменение процесса модерации вакансий, описанное выше.

Все существующие:

- подписки;
- лимиты;
- статусы;
- права пользователей;
- процессы работы

должны остаться без изменений.

---

# 15. Автономность Claude

Если конкретное решение не описано в данном документе, Claude должен самостоятельно принимать наиболее подходящие архитектурные, UX- и UI-решения.

Приоритеты:

- современный внешний вид;
- единый дизайн;
- качественный пользовательский опыт;
- адаптивность;
- простота использования;
- переиспользуемость компонентов;
- поддерживаемость проекта.

Если существует несколько корректных вариантов реализации, следует выбирать тот, который соответствует лучшим современным практикам разработки SaaS-продуктов.

---

# 16. Критерии приёмки

Работа считается завершённой, если:

- выполнен полный редизайн всех страниц;
- выполнен редизайн всех компонентов;
- проект использует компоненты shadcn/ui;
- весь интерфейс полностью адаптивен;
- реализована мобильная нижняя навигация;
- реализована страница «Кабинет»;
- переработаны фильтры вакансий;
- на мобильных устройствах фильтры открываются в модальном окне;
- во всём приложении используется единый контейнер;
- карточки вакансий отображают количество просмотров и откликов;
- новые вакансии сразу отправляются на модерацию;
- после редактирования вакансии повторно отправляются на модерацию;
- главная страница полностью переработана;
- весь проект имеет современный, единообразный и профессиональный интерфейс уровня современного SaaS-продукта.
