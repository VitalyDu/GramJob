# GramJob — Список улучшений

Составлен по итогам анализа кодовой базы, документации и инфраструктуры (2026-07-17).
Приоритеты: 🔴 высокий (риск для production), 🟡 средний, 🟢 низкий.

---

## 1. Инфраструктура и эксплуатация

### 🔴 PostgreSQL backup

Продукт в production, но автоматических бэкапов БД нет. Любой сбой VPS = потеря всех данных.

- `pg_dump` по cron (ежедневно) + выгрузка в offsite-хранилище (R2/S3)
- Ротация: 7 дневных + 4 недельных копии
- Периодическая проверка восстановимости (restore-тест)

### 🔴 Мониторинг ошибок (Sentry)

Ошибки в production сейчас видны только в логах PM2 на сервере.

- Sentry для Next.js (client + server) и Strapi
- Source maps в релизном билде
- Алерты в Telegram при всплеске ошибок

### 🔴 Uptime-мониторинг

- UptimeRobot (или аналог) на `gramjob.com` и `api.gramjob.com`
- Healthcheck-эндпоинт на backend (`GET /healthz`: БД + S3 доступны)

### 🟡 Cloudflare R2 для медиа

Медиа хранятся в MinIO на том же VPS — единая точка отказа, диск не бесконечный.

- Миграция upload-провайдера на R2 + перенос существующих файлов
- CDN-раздача медиа (сейчас всё через Nginx одного сервера)

### 🟡 Smoke tests после деплоя

Автодеплой по push в main без проверки, что сайт жив после выката.

- Шаг в `deploy.yml`: curl главной, `/vacancies`, `/healthz` API после деплоя
- Автоматический откат или алерт при провале

### 🟡 Деплой не под root

`deploy.yml` заходит на серверы как `root`. Компрометация CI-секрета = полный доступ к серверам.

- Отдельный пользователь `deploy` с минимальными правами (sudo только на рестарт PM2)

### 🟢 Staging-окружение

Всё катится сразу в production. Отдельный staging (можно на том же VPS, поддомен) для проверки миграций и рискованных изменений.

## 2. Надёжность и масштабирование (техдолг)

### 🔴 In-memory счётчики → БД/Redis

Три трекера живут в памяти процесса и сбрасываются при каждом рестарте (комментарии «replaced in Sprint 6» остались, но замена не сделана):

- `vacancy/controllers/vacancy.ts` — unique-view tracker (uniqueViews завышаются после рестарта)
- `vacancy/services/credit-service.ts` — дневной лимит бустов (рестарт = лимит обнулён, можно бустить заново)
- `application/services/apply-credit-service.ts` — дневной лимит откликов (то же самое)

Варианты: таблица в PostgreSQL (просто, достаточно на текущем масштабе) или Redis (если появится второй инстанс). Дневные лимиты — это прямая монетизация, их обход через рестарты/краши стоит денег.

### 🔴 Атомарность кредитных лимитов

`checkAndConsumeVacancyCredit` считает и списывает без транзакции — race condition позволяет превысить лимит параллельными запросами.

- `SELECT ... FOR UPDATE` или атомарный `UPDATE ... WHERE credits > 0 RETURNING`

### 🟡 Rate limiting: memory driver

`koa-ratelimit` с `driver: 'memory'` — лимиты сбрасываются при рестарте и не работают при горизонтальном масштабировании. Плюс `id: ctx.ip` — за Nginx нужно убедиться, что берётся реальный IP из `X-Forwarded-For` (`app.proxy`), иначе все клиенты делят один лимит.

- Проверить проксирование IP сейчас (потенциальный баг в production)
- Redis-driver — когда появится второй инстанс

### 🟡 Очередь для Telegram-уведомлений

`sendNotification` шлёт в Telegram синхронно из lifecycle-хуков. Нет retry при сбое Bot API, нет учёта лимита Telegram (30 msg/sec). При массовых событиях (cron истечения подписок) возможны потери.

- Простая очередь: таблица `notification_outbox` + cron-обработчик с retry/backoff
- Либо BullMQ, если появится Redis

### 🟢 FTS: лимит 10 000 результатов

Известное ограничение поиска. На текущем объёме не критично, но при росте каталога — пагинация глубже отсечки ломается. Долгосрочно — Meilisearch (уже в roadmap).

## 3. Безопасность

### 🔴 Секреты в docker-compose

`POSTGRES_PASSWORD: secret`, `MINIO_ROOT_PASSWORD: secret123` захардкожены. Для локалки терпимо, но убедиться, что production-конфиги не наследуют этот паттерн, и вынести в `.env` даже локально (единообразие).

### 🟡 Аудит зависимостей в CI

- `pnpm audit` (или Dependabot/Renovate) в CI-пайплайн — сейчас уязвимые зависимости никто не отслеживает
- Автообновление патч-версий через Renovate

### 🟡 Security headers на frontend

Проверить/добавить в Next.js (или Nginx): `Content-Security-Policy`, `X-Frame-Options` (с учётом встраивания в Telegram — нужен `frame-ancestors` для web.telegram.org), `Referrer-Policy`, `Permissions-Policy`.

### 🟡 CSRF-защита

В требованиях заявлена CSRF Protection. JWT в header её в основном закрывает, но стоит формально проверить: нет ли cookie-based флоу (Telegram Login Widget callback), где CSRF реален.

### 🟢 Регулярный re-audit RBAC

После инцидента с `user.update` (коммит 07a0dea) — добавить integration-тест-матрицу «роль × эндпоинт», чтобы регрессии прав ловились автоматически, а не ручным QA.

## 4. Производительность

### 🟡 Кэширование справочников

`GET /industries` (12 отраслей + 87 специализаций) и планы/пакеты читаются из БД на каждый запрос. Дешёвая победа: in-process кэш с TTL или `Cache-Control` заголовки.

### 🟡 Индексы под фильтры каталога

Проверить `EXPLAIN` на основных запросах каталога (`/vacancies` с фильтрами workFormat/employmentType/seniority + сортировки boost/top): составные индексы под частые комбинации. GIN для FTS есть, btree под фильтры — проверить.

### 🟢 Аналитика: рост таблиц

`VacancyAnalytics`/`ResumeAnalytics` растут на запись в день на сущность. Заложить политику ретенции (агрегация в месячные записи после N дней).

## 5. Качество кода и тестирование

### 🟡 E2E-тесты (Playwright)

451 + 271 unit/integration тестов, но ни одного E2E. Критичные флоу — регистрация, создание вакансии, отклик, оплата (мок) — проверяются только вручную (QA-чеклисты в `docs/qa/`).

- Playwright: 5–7 smoke-сценариев на golden path
- Запуск в CI на PR (против локального Strapi + PG, инфраструктура в CI уже есть)

### 🟡 Тестовое покрытие оплаты

Telegram Stars webhook (`/telegram/webhook`) — самый денежный код. Убедиться, что есть integration-тесты на: повторную доставку `successful_payment` (идемпотентность), невалидный webhook secret, несовпадение payload.

### 🟢 Легаси-зачистка

- `draft` в enum статусов (vacancy/resume/company) — legacy после авто-модерации; спланировать удаление с миграцией
- Комментарии «replaced in Sprint 6» на in-memory трекерах — вводят в заблуждение, обновить при исправлении п. 2

### 🟢 forceExit в integration-тестах

Коммит 7702bf7 добавил `forceExit` из-за гонки event hub Strapi — это маскировка незакрытых хендлов. При обновлении Strapi проверить, можно ли убрать.
